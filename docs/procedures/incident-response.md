# Incident Response Procedure

**Document ID:** D60-9-IRP  
**Version:** 1.0  
**Date:** 2026-07-09  
**Standard:** ISO 27002 5.24/8.32, ISO 20000-1 8.5.1  
**Owner:** Information Security Manager (ISM)

## Purpose

Define a structured approach to detect, contain, eradicate, and recover from security incidents affecting the One Window Bayanihan system.

## Scope

This procedure covers all security incidents impacting:
- DMW Region VII case management system
- Associated infrastructure (Render, Supabase, Cloudinary, OpenAI)
- User data (OFWs, employers, agencies, DMW staff)
- Third-party integrations

## Incident Severity Levels

| Level | Label | Example | Response Time |
|-------|-------|---------|---------------|
| P1 | Critical | Data breach, service outage >1h | <15 min |
| P2 | High | Unauthorized access, persistent DoS | <1 h |
| P3 | Medium | Suspicious activity, non-critical vulnerability | <4 h |
| P4 | Low | Policy violation, minor misconfiguration | <24 h |

## Incident Response Phases

### 1. Detect & Report

All team members are responsible for reporting suspected incidents:
- **Internal detection:** Application logs (Sentry), monitoring alerts, automated security scans
- **External report:** DMW IT, partner agency, user report via helpdesk
- **Reporting channel:** #security Slack channel or email to ISM

**Checklist:**
- [ ] Record date, time, reporter, and initial observations
- [ ] Assign severity level
- [ ] Open incident tracking issue in GitHub with label `security-incident`
- [ ] Notify ISM and relevant team members

### 2. Contain

Immediate actions to limit damage:
- [ ] Isolate affected systems (disable user accounts, revoke API keys, block IPs)
- [ ] Preserve logs and forensic evidence (snapshot instances, copy logs)
- [ ] Apply emergency patch if vulnerability is known
- [ ] Rotate compromised credentials (48-hour SLA for P1/P2)
- [ ] Document containment actions with timestamps

### 3. Eradicate

Remove the root cause:
- [ ] Identify root cause (log analysis, code review, infrastructure audit)
- [ ] Apply permanent fix (code patch, config change, access revoke)
- [ ] Verify fix in staging environment
- [ ] Scan for indicators of compromise (IoC) across the system

### 4. Recover

Restore normal operations:
- [ ] Restore from clean backup if data was compromised
- [ ] Deploy fix to production via standard CI/CD pipeline
- [ ] Monitor system for 48 hours post-restoration
- [ ] Confirm all services operational via health endpoint (`/up`)
- [ ] Re-enable any disabled accounts or services

### 5. Post-Mortem

Conduct within 5 business days for P1/P2, 10 business days for P3/P4:
- [ ] Document timeline of events
- [ ] Identify contributing factors
- [ ] Define corrective actions
- [ ] Update risk register and SoA
- [ ] Share lessons learned (anonymized) with team
- [ ] Close incident tracking issue with `completed` state

## Communication

### Internal
- P1/P2 incidents: Immediate notification to ISM, technical lead, and DMW IT contact
- P3 incidents: Daily status updates during active response
- P4 incidents: Summary in weekly standup

### External
- DMW management: P1 incidents within 1 hour, P2 within 4 hours
- Affected data subjects: As required by RA 10173 (NPC notification within 72 hours)
- Partners (Render, Supabase, Cloudinary): As applicable per DPA terms

## Incident Register

Incidents are tracked in GitHub Issues with the following metadata:
- Incident ID (auto-generated by `IncidentIdService`)
- Severity (P1-P4)
- Status (Open / Containing / Eradicating / Recovering / Closed)
- Detection date
- Resolution date
